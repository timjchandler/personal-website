Setup

Here we will download and setup the NetLogo software.

Downloading NetLogo

NetLogo is available for free here
I would recommend downloading the latest version. The system requirements for this version are fairly minimal, working on Windows systems dating back to Vista, MacOS 10.8.3 and most Debian based Linux distros.
Personally, I have run it with no issues on Ubuntu 20.04 and MacOS Big Sur and Monterey.

Installation

The download should be fairly quick, with the total size being only 200-250 MB depending on the system.
Once the download is complete installation is very simple.
On MacOS you simply locate the NetLogo application file within the downloaded folder and run it.

    Loading the Base Model

    Here we will load and run the base Forest Fire model. This will help to gain some familiarity with the NetLogo interface.

    Go to File in the top menu and select Models Library. Within this expand the Sample Models tab and then Earth Science. From here select and open the Fire model.
    You will be greeted by a large black square, a density slider, setup and go buttons as well as a percent burned field.

    The model as it appears when initially loaded

    You will also notice some tabs at the top, giving you access to information about the model as well as its code. We will look more at these later.
    First, lets start some fires! In order to burn the forest, you need a forest to burn. So press the setup button.
    This will generate your forest. As I'm sure you can tell from the almost lifelike graphics, the green pixels represent trees, with the black pixels representing empty space. If you pay close attention, you might notice a red line of pixels down the left hand side. That is the initial fire.
    The placement of the trees is randomly set each time the model is set up. You can also change the ratio of trees to empty space using the density slider. For now lets leave it at its default (57% if you've already changed it).

    Running the Model

    Running the default settings

    With the model loaded and set up, and your beautiful forest planted it's time to burn it down.
    In what will probably not come as a surprise to you, the model is started by pressing go.
    You should see the fire move across from the left towards the right, most likely burning between 10% and 25% of the forest.
    It is worth running it a few times in order to get a good idea of the expected coverage. You can speed it up by adjusting the slider at the top above the ticks counter.
    Each time you want to rerun the model you have to press setup first to generate a new forest.

    Density

    As you may have guessed, a denser forest will burn more easily in this model. You can check this by adjusting the density slider (press setup again after any changes).
    This is due to the way in which fire spreads within the model. In order for an unburned tree to catch fire it must be directly adjacent to a burning tree. This only applies to trees above, below, left or right - not diagonals. As such, with lower density forests there is a higher chance of natural fire-breaks preventing the fire from spreading far.
    In the next section, we will explore creating artificial fire-breaks, amongst other things.

    Closing Thoughts

    I hope this has been an interesting initial foray into the world of NetLogo and multi-agent modelling.
    If you've enjoyed this then I would recommend considering some other modifications to the model and trying to implement them. Additionally, it could be interesting to explore some of the other default models and play around with them.

    "All models are wrong, but some are useful"

    - George Edward Pelham Box



    Of course, this is at the end of the day a model - and many important factors are ignored or overlooked for a variety of reasons.
    When designing a model it is important to consider what it is you actually want to examine. If your goal is to see how tree density effects the spread of a fire, then the default model could be quite useful. If you were interested in the efficacy of fire-breaks on forest fires then the model would likely be less useful, since under the existing setup a firebreak will always work, whereas in reality numerous factors such as wind, and smaller foliage could cause a fire-break to fail.
    This is in no way meant to put you off using models - but rather to consider what you are actually modelling, and to check it is analogous to what you want to measure.

    Other Models

    There are a lot of models available in the default library. As such, there is a decent chance that there is an existing model related to an area you are already interested in.
    If you're unsure of what to try though, I would recommend having a look at one of the following:
    Sample Models/Biology/Wolf Sheep Predation: This model provides a predator/prey simulation in which there are two species, wolves and sheep that both need food to survive and reproduce. The wolves kill and eat the sheep, the sheep eat grass, and the grass regrows over time. If there are too many wolves, the sheep will go extinct as they are all eaten and the wolves will follow shortly afterwards as they starve. If there are not enough wolves, the sheep will overpopulate and begin to run out of grass.
    Sample Models/Networks/Virus on a Network: This model generates a graph in which the nodes represent individuals and the edges represent connections. A virus then spreads through the graph with nodes starting susceptible to it and becoming infected or resistant over time. Personally, I have previously worked on a project in which we modified this model to create a gossip network and examine the dissemination of fake news.

    Don't Start From Scratch

    Whilst it might be tempting to build a model up from nothing it can often be much quicker and easier to adapt an existing model.
    Of course, if want to make a model simply to make a model, then don't let me stop you. But if you are trying to simulate a specific phenomena, then using an existing model as a basis can save you a lot of time and debugging, particularly if you are not overly familiar with NetLogo.
